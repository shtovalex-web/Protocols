# Программа для формирования протоколов проверки знаний по охране труда

Настольное приложение на **Python 3** и **tkinter**: загрузка сотрудников из Excel, выбор программ обучения (листы B, V_PROF, PP, SIZ, V), заполнение шаблона протокола **Word (.docx)** или текстового бланка, предпросмотр, сохранение в **PDF** / **DOCX**, журнал в **SQLite**, выгрузка данных для **Минтруда / АКОТ**.

## Требования

- **Python 3.10+** (рекомендуется 3.11–3.12; для сборки `.exe` через PyInstaller на очень новых версиях Python колёса могут отсутствовать).
- В установке Python должен быть компонент **tcl/tk** (окна интерфейса).
- **Windows** (пути, Word для PDF, реестр Минтруда ориентированы на типичную среду пользователя).

## Установка зависимостей

Из корня проекта:

```bash
py -3 -m pip install -r requirements.txt
```

Для PDF с оформлением как в Word на **Windows** нужен установленный **Microsoft Word** и пакет **pywin32** (обычно уже стоит вместе с зависимостями; иначе `py -3 -m pip install pywin32`). Конвертация идёт через Word COM, без `docx2pdf` (у последнего прогресс-бар ломается при `sys.stderr is None` в оконном приложении).

На других ОС при необходимости можно использовать `docx2pdf`. Упрощённый PDF без Word — из текста предпросмотра.

## Запуск

```bash
py -3 main.py
```

Рабочий каталог — папка с `main.py` (в разработке шаблоны в `bundle/`); при сборке `.exe` комплект лежит в **`data/`** рядом с exe, рабочие файлы — в корне с exe (см. `app_paths.py`).

## Файлы рядом с программой

| Файл / папка | Назначение |
|--------------|------------|
| `data/` (только у `.exe`) или `bundle/` (исходники) | Шаблоны Word, образцы Excel, XSD Минтруда, `FAQ.txt`, инструкции `.docx`. |
| `default_protocol.docx` | Шаблон протокола Word (в `data/` / `bundle/`). |
| `Data_base.xlsx` | База сотрудников (рабочая копия — в корне с программой; образец — в `data/` / `bundle/`). |
| `FAQ.txt` / `FAQ.md` | Текст справки в меню «Справка». |
| `icon.ico` | Иконка окна (необязательно). |
| `Шаблон_Минтруд_XSD_УМН.xlsx` | Официальный Excel-шаблон для выгрузки в Минтруд (лист «Шаблон»). |
| `protocols.db` | Создаётся автоматически: журнал протоколов и настройки. |
| `last_protocol_no.json` | Последний введённый № протокола. |
| `protocol_errors_journal.txt` | Журнал ошибок (UTF-8): сообщения из окон «Ошибка» и трассировки необработанных сбоев. |
| `Protokol/` | По умолчанию открывается при сохранении протокола в PDF/DOCX. |
| `Mintrud/` | По умолчанию при сохранении шаблона выгрузки Минтруда. |

## Основные возможности

- Формирование протокола по шаблону **.docx** / **.txt** с подстановкой даты, номера, тем, таблицы результатов, блоков Б / ПП / СИЗ / В.
- Подстановка номеров из файла **«Реестр обученных…»** (СНИЛС/ФИО, протокол, программа); строгое совпадение по должности — **только для программы «Б»** (см. `FAQ.md`).
- **Минтруд**: реквизиты работодателя, выгрузка Excel по журналу; при сохранении создаются **два файла** с одним именем: `.xlsx` и **`.xml`** (те же строки данных, удобно для контроля и обработки).
- **Комиссия** и родительный падеж для подписей (при установленных `pymorphy2` и словарях).
- **Администрирование → Выгрузка шаблонов восстановления** — папка с пустым `protocols.db`, шаблоном `Data_base.xlsx` (листы сотрудников и комиссии), `last_protocol_no.json`, копиями `default_protocol.docx`, FAQ/README, шаблонов Минтруда (если есть в комплекте) и файлом `ВОССТАНОВЛЕНИЕ_ДАННЫХ.txt` с инструкцией — на случай утери или порчи основных файлов.
- Подробности по полям шаблона — кнопка «Переменные шаблона» в окне настроек и файл `bundle/FAQ.md`.

## Эталонная копия для переноса на другие ПК

Скрипт собирает в папку `эталон_сборки` актуальные `.py`, `README.md`, `requirements.txt` и файлы из `bundle/` (шаблоны Word, FAQ, XSD Минтруда и т.д. — в эталоне они окажутся рядом с `main.py`), плюс `запуск.bat`, `установка_зависимостей.bat`, `ИНСТРУКЦИЯ.txt`.

```bash
py -3 tools/update_etalon.py
```

В проводнике Windows можно запускать **`update_etalon.bat`** в корне проекта (окно консоли не закроется сразу).

После изменений в проекте скрипт стоит запускать повторно.

## Сборка исполняемого файла (.exe)

Нужен **PyInstaller** (см. `requirements-build.txt`):

```bash
py -3 -m pip install -r requirements-build.txt
py -3 build_windows_exe.py
```

Либо двойной щелчок по **`build_windows_exe.bat`** (папка вывода по умолчанию та же; путь к другой папке можно передать аргументом или перетащить папку на `.bat` в проводнике).

Результат: папка **`ProtocolOHT_onefile`** — **`ProtocolOOT.exe`**, подпапка **`data/`** (шаблоны, `FAQ.txt`, образцы Excel, XSD Минтруда; без `.md`). Переносите **exe и `data/`** целиком. Краткая памятка — `ИНСТРУКЦИЯ_папки_сборки.txt` в корне папки сборки.

Для PDF «как в Word» на целевом ПК нужны **Microsoft Word** и корректная работа **COM** (pywin32 входит в exe).

## GitHub: синхронизация изменений

Репозиторий: `https://github.com/shtovalex-web/Protocols.git` (ветка `main`).

**Ручная синхронизация** (pull → commit → push):

```bash
py -3 tools/sync_github.py -m "описание изменений"
```

Или **`sync_github.bat`** в корне проекта.

**Автоматический push после каждого локального commit** (один раз на этом ПК):

```bash
setup_git_hooks.bat
```

После этого каждый `git commit` вызывает `git push origin <ветка>`. При сбое сети повторите `sync_github.bat`.

На GitHub при push/pull request запускается проверка `tools/verify_project.py --no-launch` (см. `.github/workflows/verify.yml`).

Ветка **`linux`**: полная копия для Linux, синхронизация с `main` — `python tools/sync_linux_branch.py` или автоматически через CI (`sync-linux.yml`).  
**Бинарник PyInstaller** (дистрибутив без исходников): `docs/LINUX_DIST.md`, сборка в CI — workflow `build-linux-dist`.

Подробнее: `docs/GITHUB_SYNC.md`.

## Разработчик

Сведения для пользователей задаются в `ProtocolOHT_next/protocol_app_info.py`:

- `APP_VERSION` — строка версии; **журнал изменений** — `bundle/ЖУРНАЛ_ДОРАБОТОК.md` (ведётся с 1.4, открывается из меню «Справка»);
- `APP_DEVELOPER_NAME` — ФИО или наименование;
- `APP_DEVELOPER_ROLE` — кратко, чем занимался автор (необязательно);
- `APP_DEVELOPER_ORG` — организация (необязательно);
- `APP_DEVELOPER_CONTACT` — email, телефон или другой контакт (необязательно).

В окне программы они отображаются в меню **«Справка» → «О программе…»**. Если константы пустые, в диалоге показывается подсказка, где их заполнить.

## Лицензия и поддержка

Проект прикладной; условия распространения задаёт владелец репозитория. Актуальные требования портала Минтруда / АКОТ уточняйте по официальным инструкциям — программа формирует файлы по заложенной логике и шаблонам из комплекта.
